# Qraft リリース & バージョン管理ガイド

Qraft プロジェクトにおける、バージョン番号（タグ）の付け方とリリース手順のガイドラインです。

## 1. バージョン番号のルール (Semantic Versioning)

バージョンは `vX.Y.Z` の形式で管理します。

*   **メジャー (X)**: **プロジェクトのマイルストーン完了**
    *   大規模な機能追加、UIの全面刷新、破壊的な変更など、プロジェクトの節目でカウントアップします。
    *   例: `v1.0.0` → `v2.0.0`
*   **マイナー (Y)**: **新機能の追加・バグ修正**
    *   通常の機能追加や、複数のバグ修正をまとめてリリースする際にカウントアップします。
    *   例: `v1.0.0` → `v1.1.0`
*   **パッチ (Z)**: **緊急の修正・微調整**
    *   リリース直後のタイポ修正や、単一の小さなバグ修正などの際に使用します。
    *   例: `v1.1.0` → `v1.1.1`

---

## 2. リリース手順 (本番デプロイのトリガー)

本番環境（Production）へのデプロイは、GitHub で **「Release」を公開した時のみ** 実行されます。

1.  **PR のマージ**: `develop` から `main` ブランチへプルリクエストを作成し、マージします。
2.  **Release の作成**: GitHub リポジトリの右側にある「Releases」から「Create a new release」をクリックします。
3.  **タグの入力**: 適切なバージョン（例: `v1.1.0`）を入力し、「Create new tag」を選択します。
4.  **ノートの生成**: **「Generate release notes」ボタン**（無料機能）をクリックします。
    *   これにより、前回のリリース以降にマージされたプルリクエストのタイトルが自動でリストアップされます。
5.  **公開 (Publish)**: 「Publish release」ボタンをクリックします。
    *   **この瞬間に、GitHub Actions が起動し、本番環境へのデプロイが開始されます。**
6.  **自動ラベル付与 (NEW)**:
    *   リリースが公開されると、GitHub Actions が自動的に起動します。
    *   リリースノートに含まれるプルリクエストに関連付けられた全ての Issue に、このバージョン（例: `v1.1.0`）のラベルを自動で付与します。
    *   これにより、Issue の一覧から「どの課題がどのバージョンで解決されたか」が一目で分かります。

---

## 3. なぜこのフローにするのか？

*   **安全性の確保**: `develop` ブランチ（Staging）で十分にテストした「確定版」だけを本番に出せます。
*   **履歴の可視化**: 「いつ、どの機能を追加したか」が GitHub 上にリリースノートとして綺麗に残ります。
*   **明確な意思決定**: 「マージ＝リリース」ではなく、「リリースノートを書いて公開する」という明確な意思をトリガーにすることで、誤デプロイを防ぎます。

# 開発ワークフローガイド

本プロジェクトでは、品質とスピードを両立させるため、以下のプルリクエスト・ベースの開発フローを採用しています。

## 🏹 基本サイクル

1.  **Issue の確認**: [GitHub Issues](https://github.com/junichi-muraoka/ut-qms/issues) から着手するタスクを選びます。
2.  **ブランチ作成**: 作業用のブランチを `main` から作成します。
    - 命名規則: `feature/チケット名(英語)` または `fix/バグ名`
    - 例: `git checkout -b feature/vitest-setup`
3.  **実装 & テスト**:
    - コードを実装し、それに対応する単体テストを `Vitest` で作成します。
    - 手元のターミナルで `npm run test` が通ることを確認します。
4.  **プルリクエスト (PR)**:
    - GitHub 上で `main` ブランチに対して PR を作成します。
    - **自動チェック (CI)**: GitHub Actions が自動で起動し、Lint / Type Check / Unit Test を実行します。
5.  **レビュー & マージ**:
    - 自動チェックがすべてパス（Green）したことを確認します。
    - 必要に応じてコードレビューを受け、修正します。
    - 問題がなければ `main` にマージします。
6.  **自動デプロイ (CD)**:
    - `main` にマージされると、自動的に Cloudflare の本番環境へデプロイされます。

## ✅ チケット（Issue）の書き方
新しくタスクを作成する際は、提供されているテンプレートに則り、以下の項目を必ず含めるようにしてください：
- **背景・目的**: なぜやるのか
- **完了条件**: 何ができれば OK か
- **詳細タスク**: 具体的な手順

## 🛡️ 品質ガード
`main` ブランチには、以下の「ガード」がかけられています：
- 自動テスト（GitHub Actions）が失敗しているコードはマージできません。
- (推奨) コードレビューで承認されない限りマージできません。

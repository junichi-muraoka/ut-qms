## 🧪 究極の検証計画 (The UTS: Ultimate Testing Standard)

### 検証項目【TC-DB-01】：WBS 自己参照（親子構造）の論理整合性
1. **検証目的**: 親タスクを削除した際、または子タスクを登録した際、DB レベルで正しい親子関係が保持・制約されることを証明する。
2. **前提条件**: `test_items` テーブルが空、または既知のテストデータのみの状態。
3. **実施手順**: 
   - ① 親レコード(A)をインサート。
   - ② 子レコード(B)の `parent_id` に (A)の UUID を指定してインサート。
   - ③ `db.query.test_items.findMany({ with: { children: true } })` を実行。
4. **判定基準 (『正しい』の定義)**: 
   - 取得された JSON オブジェクトの (A).`children[0].id` が (B).`id` と完全一致すること。
   - 親レコードが存在しない `parent_id` でのインサートを試みた際、`SQLITE_CONSTRAINT_FOREIGNKEY` エラーを返すこと。
5. **エビデンス（証明方法）**: 
   - **[証跡1]**：Vitest 実行時のコンソールログ（Assert 正解の緑色チェックマークのキャプチャ）。
   - **[証跡2]**：②で生成された SQLite 物理データベースの内容をダンプした JSON 指標。

### 検証項目【TC-DB-02】：分析用カラムのデータ型バリデーション
1. **検証目的**: `estimated_hours` (予定工数) や `cause_category` (原因分析) に、不適切なデータ型や未定義の区分値が入るのを防げることを証明する。
2. **前提条件**: API サーバーがテスト環境で起動しており、Drizzle ORM のバリデーションが有効。
3. **実施手順**: 
   - `estimated_hours` に文字列（"abc"）を送信して `insert` を試行。
4. **判定基準 (『正しい』の定義)**: 
   - アプリケーションレベル、または DB レベルで `Type Error` が発生し、不適切なデータが保存されないこと（エラーレスポンス `400` を確認）。
5. **エビデンス（証明方法）**: 
   - **[証跡]**：API レスポンスのエラーログ・メッセージのスクリーンショット。

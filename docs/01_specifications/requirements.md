# 要件定義書 (Requirements Definition)

## 1. プロジェクト・ビジョン (Core Vision)
Qraft（クラフト）は、ソフトウェア開発のライフサイクルにおける全データをリアルタイムに集約し、品質を科学的に管理することで、「報告書作成という名の儀式」を根絶します。
チーム全員が **「今、どこにリスクがあり、何をすべきか」** を即座に把握し、最速で品質強化（Hardening）を行える世界を実現します。

## 2. コア・バリュー (Core Values)
- **Zero Ritual**: 報告書やエビデンス集計のための残業をゼロにする。
- **Predictive Quality**: データに基づき、将来の遅延やバグ残数を予測し、先手を打つ。
- **Single Source of Truth**: 計画、ルール、進捗、会話、成果物を Qraft 一箇所に集約する。

## 3. 機能要件 (Functional Requirements)

### A. プロジェクト推進 & 体制
- **WBS & スケジュール (Gantt)**: 階層構造による進捗ロールアップと、リアルタイムな納期影響予測。
- **マイルストーン管理**: 各フェーズの品質ゲート基準（判定条件）の定義と自動判定。
- **体制図 (Org Chart)**: プロジェクトメンバーの役割と担当モジュールの可視化。

### B. 高度な品質管理 (Advanced QMS)
- **リアルタイム・メトリクス**: テスト密度、バグ密度、バグ収束曲線の常時算出。
- **未然防止（レビュー）**: リアルタイム議事録・レビュー記録表の作成と指摘内容分析。
- **再発防止（バグ分析）**: 原因区分、バグ種類の分類と根本原因（Root Cause）の特定。
- **トレーサビリティ**: 要件 → 設計 → テスト → バグ の完全な追跡。

### C. エビデンス・知識管理 (Artifact & Knowledge)
- **マルチ・エビデンス Hub**: 複数画像、ログ、動画等のテスト証跡を統合管理。
- **成果物一覧 (Artifacts)**: 納品に必要なドキュメントの所在と完成度の一元管理。
- **プロジェクト Wiki**: 計画書、ルール、議事録の Markdown ベースの共有基盤。

### D. コミュニケーション (Collaboration)
- **コンテキスト・コメント**: チケット（課題、不具合）上での直接議論。
- **トピック別チャネル**: Slack ライクなリアルタイム・メッセージング。

### E. オート・レポータ (Auto Reporting)
- **週次報告書 (Weekly)**: 定量データ（実績）と定性評価（コメント・リスク）の自動統合。
- **品質分析報告書 (Full)**: 全データ、グラフを包含した公式提出用ドキュメントのワンクリック出力。

### F. プログラム管理 (Multi-System / PMO Support)
- **マルチシステム・ポートフォリオ**: 複数システムの進捗・品質を横断俯瞰する「管制塔（Tower）」ビュー。
- **システム間依存関係 (Dependency)**: 後続工程の開始条件となる「他システムのマイルストーン」の可視化。
- **共通品質ゲート**: 組織全体の標準に基づいた、システム横断的な品質アラート（PMO 向け抽出）。

## 4. 非機能要件 (Non-Functional Requirements)
- **エッジネイティブ・リアルタイム**: Cloudflare Workers/D1 による多人数同時アクセスの低レイテンシ化。
- **セキュリティ**: Google OAuth 2.0 による統合認証。

## 🛠 開発・品質管理方針

プロジェクトの「真の品質」を担保するため、以下の開発サイクルを徹底する。

### 1. 統合型 Issue 運用 (Single Source of Truth)
全ての開発タスクは、GitHub Issue 上で以下の 2 点が定義された状態で開始される。
- **実装設計 (Implementation Design)**: テーブル定義、ロジック、データ型等の詳細技術仕様。
- **検証実施要領 (Verification Table)**: Unit/E2E テストの種別、手順、および客観的な判定基準。

### 2. 検証の自動化と客観性
- サーバー内部論理は **Vitest (Unit)**、ユーザー体験は **Playwright (E2E)** による完全自動テストで検証する。
- 判定基準は「主観」を排し、物理的な事象（SQL結果、エラーコード、DOMの状態）で定義する。

### 3. エビデンス（証跡）添付の義務化
- 全ての検証項目に対し、実行ログやスクリーンショット等の物理的な「証拠」の Issue コメント添付を必須とする。
- 定義された「判定基準」と「エビデンス」が 100% 一致しない限り、タスクは完了（Done）と見なさない。
